# AI Flight Lab 多系统适配方案

更新时间：2026-06-27

适用项目：姜睿个人网站 / AI Flight Lab

当前框架：Vite + Vue 3 + TypeScript + Vue Router Hash History

线上地址：https://999888666jiang-dev.github.io/jiangrui-ai-flight-lab/

GitHub 仓库：https://github.com/999888666jiang-dev/jiangrui-ai-flight-lab

## 1. 目标

本方案用于保证 AI Flight Lab 在桌面端、平板、手机、iOS Safari、Android Chrome、微信内置浏览器、PWA 和后续原生封装中稳定展示，并尽量保留当前高级动效体验。

核心目标不是简单“能打开”，而是同时满足：

- 首页 Three.js 透明科技桨叶、全局粒子、两侧滚轮粒子、中区精密粒子保持高级感。
- Video Bay、FPV 飞行视频、成交成果展示可以播放真实视频。
- Evidence Vault、6 个展示详情页、2 个媒体集合页都能直达。
- 移动端不横向溢出，导航、标题、按钮、二维码、执照、视频不遮挡。
- 微信、iOS、低性能设备可以自动降级，不出现空白页或卡死。
- 后续素材更新主要改资源目录、manifest 和 Release 上传，不重写页面结构。

## 2. 总体策略

采用渐进增强路线：

1. 响应式网页优先。
2. PWA 作为推荐增强层。
3. GitHub Pages 发布网站源码。
4. GitHub Releases 托管大视频。
5. Capacitor 原生封装只作为后续可选阶段，不提前重写架构。

不采用 uni-app、Flutter 或多端重写。原因是当前项目的核心价值在 Web 视觉、Canvas、Three.js、视频、Hash Router 和 GitHub 静态发布，重写会增加维护成本，也容易破坏已有动效系统。

## 4. 统一能力配置

后续页面和组件不得各自直接判断 iOS、微信、低性能设备。统一从 composable 获取策略。

核心配置类型：

```ts
type DeviceTier = 'high' | 'medium' | 'low' | 'minimal';
type RuntimeShell = 'desktop' | 'mobile-browser' | 'wechat' | 'pwa' | 'capacitor';

type MotionBudget = {
  particleFactor: number;
  canvasFps: number;
  canvasDpr: number;
  three: boolean;
  scanLines: boolean;
  hoverMotion: boolean;
  compositeOperation: 'lighter' | 'source-over';
};

type VideoPolicy = {
  autoplay: boolean;
  mutedRequired: boolean;
  gestureRequired: boolean;
  preload: 'none' | 'metadata' | 'auto';
};
```

设计原则：

- 桌面高性能设备启用完整 Three.js、Canvas 粒子、滚动动效和 hover 交互。
- 平板降低粒子数量和 WebGL 像素比。
- 手机保留视觉特征，但减少粒子密度、扫描线和持续动画成本。
- 微信环境默认极轻降级，优先保证内容可读和视频可手势播放。
- `prefers-reduced-motion` 下关闭强扫描、强吸入、持续高频粒子，只保留低透明静态氛围。

## 5. 视频适配策略

### 播放源策略

开发环境：

- 优先使用本地 `public/videos/` 和 `public/media/`。
- 方便继续预览新增素材。

生产环境：

- 使用 GitHub Releases URL。
- 固定 Release tag：`media-current`。
- URL 格式：

```text
https://github.com/999888666jiang-dev/jiangrui-ai-flight-lab/releases/download/media-current/<assetName>?v=<fileVersion>
```

### 视频播放策略

- 默认 `muted`。
- 使用 `playsinline`、`webkit-playsinline`、`x5-video-player-type="h5"`。
- 默认 `preload="metadata"`，列表页不预加载全部正片。
- 声音开关只影响当前播放器。
- 自动播放失败时显示点击播放遮罩。
- 如果浏览器已经实际开始播放，播放遮罩必须自动消失。
- Service Worker 不缓存 MP4 和 Release 下载 URL。

### 当前公开视频资产

- Video Bay 背景视频：
  - `video-bay__fpv-lab-background.mp4`
- FPV 飞行视频：
  - `fpv-flight-video__fpv-001.mp4` 至 `fpv-flight-video__fpv-011.mp4`

## 6. 页面适配规则

### 全局布局

- 顶部导航、任务侧轨、全屏视频、灯箱、底部区域必须使用 safe-area 变量。
- fixed 元素不得硬编码贴边距离。
- 所有全屏层必须 `pointer-events: none`，除非它是明确交互控件。
- 粒子层级必须低于导航、按钮、二维码、证照、灯箱控制。

### 首页

- 桌面端保留 Three.js 透明科技桨叶。
- 移动端可关闭 Three.js，仅保留 CSS/Canvas HUD 氛围。
- Hero、任务、能力系统、Mission Control、联系区都必须检查横向溢出。

### Video Bay

- 桌面端显示真实视频背景与 HUD。
- 移动端视频可铺满或裁切，但不得压住正文按钮。
- 视频播放失败时显示点击播放兜底。
- 声音按钮必须可点击，且不被底部安全区遮挡。

### Evidence Vault

- 执照、二维码和 PDF 入口优先保证清晰可读。
- 粒子和扫描线不得遮挡二维码。
- 6 张卡片进入详情页时保留吸入动效；低动态偏好下改为淡入直达。

### 展示详情页

- 6 个展示页都可直达。
- FPV 与成交成果使用真实媒体舞台。
- 其他页面继续使用真实素材占位，不请求不存在资源。
- 旧路由 `/#/evidence-vault/code-projects` 保留重定向到成交成果展示。

### Reel 媒体集合页

- FPV 使用回忆胶卷 / 飞行胶片滚动体验。
- 成交成果使用流光展柜 / 聚光灯播放器。
- 卡片默认不加载完整视频，点击后再进入播放器。
- 灯箱关闭后应回到原页面上下文，不丢滚动位置。

## 7. GitHub 发布维护流程

### 常规源码更新

```powershell
npm run build
.\scripts\sync-github.ps1 -Message "update site"
```

### 新增或替换视频

1. 把视频放入对应本地目录：
   - `public/videos/`
   - `public/media/fpv-flight-video/`
   - `public/media/deal-results-showcase/`
2. 运行上传脚本：

```powershell
.\scripts\upload-release-media.ps1 -Tag media-current
```

3. 如果 manifest 发生变化，提交源码：

```powershell
.\scripts\sync-github.ps1 -Message "update release media manifest"
```

4. 等待 GitHub Actions 成功。
5. 访问线上页面复测。

### 仓库规则

- `.mp4`、`.mov`、`.m4v`、`.webm` 不提交到 Git。
- `.release-media/` 不提交。
- `.tools/` 不提交。
- `个人信息资源包/` 不提交。
- 视频更新通过 Release 覆盖同名 asset，网页 URL 保持稳定。

## 9. 风险与规避

### 风险 1：iOS 或微信自动播放失败

规避：

- 默认静音。
- 使用 inline 播放属性。
- 保留点击播放遮罩。
- 用户点击后通过 `requestPlayback` 解锁。

### 风险 2：低端手机粒子或 Three.js 卡顿

规避：

- 统一 `DeviceTier` 分级。
- 低性能关闭 Three.js。
- 降低 Canvas DPR、FPS 和粒子密度。
- 微信使用 minimal profile。

### 风险 3：GitHub Pages 不适合大视频

规避：

- MP4 不放 Pages。
- MP4 放 GitHub Releases。
- 页面生产环境使用 Release URL。
- 后续访问量高时可迁移到对象存储或 CDN，manifest 结构保留扩展位。

### 风险 4：Service Worker 缓存爆配额

规避：

- MP4 和 `/releases/download/` 请求不缓存。
- 只缓存页面壳、JS、CSS、轻量图片和必要静态资源。

### 风险 5：新增素材路径包含中文导致 404

规避：

- 公开资源使用安全 ASCII 文件名。
- manifest 保留原始中文文件名用于展示。
- 页面只请求 manifest 中确认存在的 `src` 或 `streamUrl`。

### 风险 6：后续页面新增动效破坏阅读

规避：

- 动效层必须低于内容层。
- 对标题、正文、按钮、二维码、执照、视频控件设置保护区。
- 先降低亮度和密度，不用继续叠加粒子解决空感。

### 风险 7：GitHub Actions 首次 Pages 启用失败

规避：

- 首次发布可用 `gh api` 手动启用 workflow Pages。
- 工作流保留 `configure-pages` 和 `deploy-pages`。
- 失败时优先查 Actions 日志，不猜测。
