## 五、🔴 高风险项详解与解决方案

### R1: 抖音视频链接过期 — 等级 20

**问题**: 参考视频1是抖音CDN临时URL（`v95-web-sz.douyinvod.com`），通常数小时至数天即过期。一旦过期，该参考素材将永久丢失。

**解决方案**:
- ✅ **立即下载**: 用户需马上用浏览器或`curl`下载该视频到本地
- 备选：使用`yt-dlp`或类似工具抓取（如果链接已过期）
- 下载后放入 `source-videos/` 目录统一管理

**时间窗口**: **必须在本项目开始前完成**

---

### R2: AV1编码器不可用 — 等级 12

**问题**: FFmpeg的AV1编码器（libsvtav1/libaom）不是默认编译选项，大多数系统的FFmpeg不包含AV1编码支持。Windows上尤其常见。

**解决方案**:
- ✅ **改用H.264 + VP9双格式**（放弃AV1）
  - H.264 baseline：100%浏览器覆盖，作为主格式
  - VP9：比H.264小30-50%，Chrome/Firefox/Edge全覆盖，Safari 14+支持
  - HTML `<video>` 用 `<source>` 标签自动降级
- 编码命令改用 `libx264` 和 `libvpx-vp9`
- 如果FFmpeg本身不可用 → 使用浏览器端压缩（见备选方案B）

**备选方案B（无FFmpeg时）**:
- 使用Node.js的`fluent-ffmpeg`或WASM版FFmpeg（`@ffmpeg/ffmpeg`）
- 或者接受原视频大小，通过`preload="none"`和懒加载减轻影响

---

### R3: 自定义GLSL Shader编译失败 — 等级 15

**问题**: 粒子proximity shader在以下情况可能编译失败：
- 移动端GPU（Mali、Adreno老型号）对某些GLSL语法支持不完整
- Safari的ANGLE层对自定义shader有额外限制
- Shader错误无console输出，表现为粒子完全不可见

**解决方案**:
1. **Shader分级策略**:
   ```
   Tier 1（桌面级GPU）: 完整自定义ShaderMaterial + proximity推斥
   Tier 2（移动端中高端）: 简化ShaderMaterial（仅波动，无proximity）
   Tier 3（低端/问题GPU）: MeshStandardMaterial + JS动画驱动位置
   ```
2. **编译检测与降级**:
   ```ts
   // 检测shader编译状态
   const shader = new THREE.ShaderMaterial({...});
   shader.onBeforeCompile = (info) => {
     if (!info.compiled) fallbackToStandardMaterial();
   };
   ```
3. **开发阶段测试**: 至少在Chrome/Firefox/Safari + 一台Android真机上测试
4. **语法保守**: 使用GLSL ES 1.00语法（`precision mediump float`），避免GLSL 3.0特性

### R4: 缺少个人照片和证书扫描件 — 等级 15

**问题**: 当前素材只有视频和PDF简历，缺少：

  - 证书 → 用CSS绘制的证书卡片，展示文字信息，注明"证书扫描件备索"
  - 项目照片 → 视频提取帧作为封面图

### R5: Three.js/R3F首屏加载体积过大 — 等级 16

**问题**: Three.js(~150KB) + R3F(~40KB) + Drei(~50KB) = ~240KB gzipped。在4G网络下需3-5秒加载。国内移动网络环境可能更慢。

**解决方案**:
1. **最大力度代码分割**:
   ```ts
   // HeroSection.tsx - 绝不在首屏同步加载Three.js
   const HeroScene = React.lazy(() => import('./HeroScene'));
   // Suspense期间显示纯CSS渐变背景 + 骨架屏
   ```
2. **Import路径优化**: 只从`three`的具名导出中导入所需模块，让Vite tree-shaking生效
3. **Vite manualChunks配置**:
   ```ts
   build: {
     rollupOptions: {
       output: {
         manualChunks: {
           three: ['three'],
           r3f: ['@react-three/fiber', '@react-three/drei'],
           gsap: ['gsap'],
         }
       }
     }
   }
   ```
4. **首屏只加载关键CSS**: Tailwind purged后 ~10-15KB
5. **目标**: 首屏交互时间（不含3D）< 2秒，3D场景延迟加载< 5秒

### R6: 微信内置浏览器兼容问题 — 等级 15

**问题**: 微信内置浏览器（X5/QQ浏览器内核）在WebGL实现、Video API、触摸事件等方面有已知问题。由于用户通过微信接收素材，其联系人和潜在雇主很可能通过微信打开网站。

**具体问题**:
- WebGL上下文可能在切后台后丢失且不触发`webglcontextlost`事件
- `video.play()`需要明确的用户手势，且`muted`属性不一定绕过限制
- `canvas`的`toDataURL`可能被阻止
- 自定义字体可能被阻止下载
- `position: fixed`在输入框聚焦时行为异常

**解决方案**:
1. **UA检测 + 降级**:
   ```ts
   const isWeChat = /MicroMessenger/i.test(navigator.userAgent);
   // 微信内：禁用3D场景，使用静态hero背景图
   // 微信内：视频延迟到用户点击才加载
   ```
2. **微信专用策略**:
   - Hero区域：显示静态海报图 + CSS动画（无WebGL）
   - 视频：显示封面图 + 播放按钮，点击新开页面播放
   - 字体：纯系统字体栈
3. **引导用户用外部浏览器打开**: 页面顶部显示"在浏览器中打开以获得完整体验"提示条
4. **JSAPI检测**: 使用`WeixinJSBridge`就绪后再初始化交互组件

### R7: iOS Safari视频自动播放 — 等级 12

**问题**: iOS Safari对`video.play()`有严格限制：
- 必须在用户手势的同一事件循环中调用
- 即使`muted`+`playsinline`也不保证自动播放
- 低电量模式下所有视频暂停

**解决方案**:
1. 所有视频默认显示poster封面图
2. 在第一个用户交互（click/scroll/touch）后再尝试预加载
3. 视频播放失败时静默降级为静态图
4. Hero区域不使用视频背景（太不可靠），使用静态图或3D场景
5. Video Showcase区域：点击卡片才播放，符合用户手势要求

### R9: 视频纹理GPU内存泄漏 — 等级 12

**问题**: VideoTexture每帧将视频帧上传到GPU纹理。3个视频同时以3D纹理播放 = 720p×3 = 持续GPU带宽消耗。移动端GPU内存有限（通常256MB-512MB共享内存）。

**解决方案**:
1. **单视频策略**: 全局同时只允许1个视频纹理活跃
2. **离开视口即释放**:
   ```ts
   useEffect(() => {
     return () => {
       video.pause();
       video.src = '';
       video.load();
       texture.dispose();
     };
   }, []);
   ```
3. **视频卡片懒加载**: 仅在卡片进入视口80%时才挂载R3F Canvas
4. **移动端不使用3D视频纹理**: 手机端VideoShowcase改用标准`<video>`标签

**问题**: GitHub Pages在国内某些地区/网络环境下访问缓慢或间歇性不可用。

**解决方案**:

2. **中期优化**:

### R12: 移动端电池消耗 — 等级 12

**问题**: 连续60fps WebGL渲染是移动设备上最耗电的操作之一。用户停留30秒以上，设备明显发热。

**解决方案**:
1. **Page Visibility API**: 页面切后台时暂停整个3D渲染
   ```ts
   useEffect(() => {
     const onVisibility = () => {
       if (document.hidden) gl.setAnimationLoop(null);
       else gl.setAnimationLoop(animate);
     };
     document.addEventListener('visibilitychange', onVisibility);
     return () => document.removeEventListener('visibilitychange', onVisibility);
   }, []);
   ```
2. **滚动离开Hero后降帧**: Hero滚出视口80% → 切换到`frameloop="demand"`，仅响应滚动事件时渲染
3. **移动端30fps**: 在移动设备上将3D渲染帧率限制在30fps
4. **静止检测**: 用户3秒内无交互 → 自动将帧率降至10fps

---

### R17: 低端设备粒子性能 — 等级 12

**问题**: 2500个GPU实例化粒子在Intel集显或低端Mali GPU上可能导致帧率骤降至15fps以下。

**解决方案**:
1. **GPU Tier检测**（使用`useDetectGPU` from drei）:
   ```
   Tier ≥3 (桌面独显): 2500粒子 + 全部效果
   Tier 2 (移动中高端/集显): 800粒子 + 简化效果
   Tier 1 (移动低端): 300粒子 + 无shader
   Tier 0 (无WebGL): 静态CSS背景
   ```
2. **性能预算监控**: 使用`stats.js`（开发模式）监控帧率，低于30fps自动降一档
3. **粒子降级路径**: InstancedMesh → 点精灵(Points) → CSS粒子（绝对定位div）→ 静态渐变

## 七、技术难点攻关方案

### 难点1: 粒子Proximity Shader（核心交互）

**挑战**: 需要将鼠标屏幕坐标转换为3D世界坐标，并通过uniform传给2500个粒子的顶点shader，每个粒子计算与鼠标的距离并沿远离方向位移。

**攻关方案**:
1. 使用Drei的`useRaycaster`或手动raycasting对隐藏平面求交
2. Shader采用增量开发：
   - Step 1: 静态粒子位置随机分布（无shader）
   - Step 2: 添加简单正弦波位移（基础vertex shader）
   - Step 3: 加入uMouse3D uniform + 距离计算
   - Step 4: 调试proximity衰减曲线（线性→平滑步进→缓出）
3. 关键代码保护：shader编译失败时catch并回退到MeshStandardMaterial
4. 用Chrome的WebGL Inspector / Spector.js调试shader

### 难点2: R3F Canvas与DOM的Z-Index层叠

**挑战**: 3D Canvas需要作为Hero区域的"背景"，DOM文字/按钮在其上方。同时需要让鼠标事件穿透Canvas到达3D场景。

**攻关方案**:
```tsx
<div style={{ position: 'relative', height: '100vh' }}>
  {/* 3D背景层 */}
  <Canvas style={{ position: 'absolute', inset: 0, zIndex: 0 }} />

  {/* DOM内容层 */}
  <div style={{ position: 'relative', zIndex: 1, pointerEvents: 'none' }}>
    <h1>姜睿</h1>        {/* pointer-events: none */}
    <button style={{ pointerEvents: 'auto' }}>View More</button>
  </div>
</div>
```
- Canvas设置`pointer-events: auto`（默认）来接收3D交互
- DOM容器设置`pointer-events: none`，只有交互元素（按钮）设为`auto`

### 难点3: 滚动驱动的3D场景过渡

**挑战**: 用户滚离Hero时，3D相机平滑后退、粒子淡出、浮动几何体散开。需要在GSAP ScrollTrigger和R3F的`useFrame`之间协调。

**攻关方案**:
- ScrollTrigger负责提供0-1的progress值
- `useFrame`中读取progress，计算相机z、粒子opacity uniform、几何体目标位置
- 使用`lerp`平滑过渡，避免突兀变化
- 关键：`useFrame`中的计算保持轻量（仅数学运算，不触碰DOM）

### 难点4: 视频压缩质量与体积平衡

**挑战**: 12MB原视频需压缩到可接受大小，但无人机航拍画面通常有大量快速运动和细节（树木、建筑），压缩后容易出现块效应。

**攻关方案**:
1. 两阶段压缩：
   - 先缩分辨率：源分辨率为1080p或4K → 缩到720p
   - 降帧率：30fps或60fps → 24fps（电影感，足够流畅）
2. CRF参数调优：
   - H.264: CRF 23-26（值越大体积越小质量越差，23是视觉无损的分界线）
   - VP9: CRF 30-33
3. 如果单文件仍>3MB → 进一步降到540p
4. 输出后人工检查：在大屏幕上播放确认无明显块效应
